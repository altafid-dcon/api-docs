Shared Conventions

External vendor

This page covers the rules that apply across all Altafid APIs so integrators do not have to re-learn the same patterns on each domain page.

Audience and scope

This guide covers the full set of Altafid vendor-facing APIs, including entity management, saved views, tenant administration, staff management, and data export. All endpoints require standard JWT bearer token authentication and the tenant context headers described below.

Base URL and environment

FieldValueNotes
Base URLhttps://api.altafid.dev.altafid.netUAT endpoint used in the current documentation set.
AuthenticationBearer tokenReturned by /api/auth/login or MFA verification flows.
Tenant contextx-tenant-uuidRequired for downstream domain APIs.

Required headers

After login, most protected APIs expect both the JWT and the tenant/user context headers below.

HeaderRequiredDescription
AuthorizationYesBearer {token} from login or MFA verification.
x-tenant-uuidYesTenant scope for the request.
x-user-emailYesThe authenticated user email from login response.
x-user-idYesUser UUID from the login response. For tenant staff, this is typically tenantStaffUuid.
x-user-typeYesUser type such as TENANT_STAFF, PARTNER_STAFF, or CONTACT.

Pagination conventions

List endpoints generally use Spring pagination.

  • page is zero-based.
  • size defaults to 20 on most endpoints.
  • sort is endpoint-specific for classic GET lists and an array for v2 filter endpoints.
GET /api/contacts?page=0&size=20
GET /api/organisations/search?searchTerm=alta&page=0&size=20

V2 filtering conventions

Contacts, organisations, and households expose v2 POST-based filtering APIs. The request accepts a list of conditions plus page, size, and sort values.

{
  "filters": [
    { "field": "isActive", "operator": "equals", "value": "true" },
    { "field": "stageName", "operator": "in", "values": ["Prospect", "Client"] }
  ],
  "page": 0,
  "size": 20,
  "sort": ["createdAt,desc"]
}

Distinct values for a filterable field are fetched through the matching /filter-values endpoint.

{
  "field": "stageName",
  "baseFilters": [
    { "field": "isActive", "operator": "equals", "value": "true" }
  ],
  "page": 0,
  "size": 20
}

Error handling

Controllers return a mix of empty responses, booleans, Spring validation failures, and structured error bodies. Vendors should code to HTTP status first and then parse the response body if present.

StatusTypical meaningRecommended action
400Validation or malformed inputFix the payload, path variable, or query parameter.
401Missing or invalid authenticationRe-authenticate or refresh the token.
403Authenticated but not allowedCheck role, tenant context, and ownership rules.
404Resource not foundVerify UUIDs and whether the resource belongs to the current tenant.
409State conflictUsed on duplicate resources or invalid transitions such as credential initialization conflicts.
Important: some endpoints return only the HTTP status with an empty body on failure, while others return a structured payload. Client code should not assume a uniform error schema everywhere.

Stable sample data guidance

  • Reuse a single documented tenant and user where possible.
  • Prefer examples that can be re-run safely, such as list, search, and filter requests.
  • For create and update samples, use obviously illustrative values and note that UUIDs returned from your tenant will differ.
  • For note attachments, treat the upload URLs as short-lived and environment-specific.
